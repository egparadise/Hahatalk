# Stage 5 Media And Document Desk

## Scope

Stage 5 replaces the metadata-only attachment demo with durable, owner-centered media storage. Text chat remains independent from upload and processing. An asset is private by default and sharing creates exact viewer grants without transferring ownership.

- Resumable multipart upload with restart recovery, bounded retries, abort, and progress.
- Whole-file and per-part SHA-256 verification before publication.
- Declared MIME, extension, and detected magic-byte comparison.
- Quarantine for executable signatures, malware fixtures, scan failure, and inconsistent payloads.
- Owner-private archive, exact conversation recipient grants, immediate revoke, trash, and restore.
- Date/place metadata, indexed owner timeline, manual albums, and GPS-stripped image derivatives.
- Authenticated PDF.js, image, video, audio, and text preview; Office files use a local fallback.
- Windows screen capture persistence and independent PDF/media pop-out windows.

## Storage Boundary

PostgreSQL stores metadata and authorization only. File bytes never enter a relational column or the static web root.

The packaged Windows app uses `LocalObjectStore` under `%APPDATA%/HahaTalk/objects`. Electron supplies that root to the bundled API. The `ObjectStore` interface owns temporary parts, assembly, immutable originals, derivatives, ranged reads, and deletion so a managed S3 implementation can replace the local provider without changing media policy.

Object keys are generated by the server. Client file names and paths are never accepted as storage keys or returned to another viewer.

## Data Model

- `media_upload_sessions`: owner/idempotency key, expected bytes/hash, part geometry, expiry, source, and completion state.
- `media_upload_parts`: exact part number, server object key, byte count, and SHA-256.
- `media_assets`: owner, immutable original key, detected type, size/hash, processing state, archive scope, private metadata, and soft deletion.
- `media_variants`: immutable shared preview or later derived representation.
- `message_attachments`: message-to-asset link with independent revocation.
- `media_grants`: exact asset/message/grantee authorization and preview/download capabilities.
- `media_albums` and `media_album_items`: owner-only organization.
- `media_processing_events`: minimal stage/status history without file bytes or private GPS.

Owner timeline indexes cover `(owner_id, captured_at, created_at, id)`, local capture time, normalized place, and archive scope. Migration `005_media_document_desk.sql` is the executable schema contract.

## Upload And Inspection

1. `POST /media/uploads` validates file name, declared MIME, source, size, optional final hash, and creates an expiring session.
2. `PUT /media/uploads/:uploadId/parts/:partNumber` streams a bounded part, verifies `X-HahaTalk-Part-Sha256`, and records an idempotent checkpoint.
3. `POST /media/uploads/:uploadId/complete` locks the session, verifies every part, assembles and fsyncs the temporary object, recomputes the final byte count/hash, and inspects the bounded payload.
4. A clean result is moved to an immutable original key. Shareable images receive a metadata-stripped preview. A blocked result remains quarantined and cannot be previewed or shared.
5. Completion is idempotent. Missing or externally quarantined parts fail closed with a re-upload response.

The standalone scanner checks signatures and test fixtures. Production external exchange requires the ClamAV adapter; scan error or skipped content must never be recorded as clean.

## Authorization

- The owner can list, preview, download, organize, update private metadata, trash, restore, share, and revoke an owned clean asset.
- A recipient can access only an active `media_grants` row tied to a delivered message.
- Hub grants are generated from the delivery snapshot, not submitted recipient IDs. A spoke participant never receives another participant, grant, or owner-library row.
- Guest download follows the conversation policy and can be false while preview remains true.
- Every content request repeats current authorization. Range requests do not bypass revoke.
- Revoking a message share immediately revokes grants, deliveries, attachment projection, and realtime visibility while preserving the owner's original.

## API

- `POST /media/uploads`
- `GET /media/uploads/:uploadId`
- `PUT /media/uploads/:uploadId/parts/:partNumber`
- `POST /media/uploads/:uploadId/complete`
- `DELETE /media/uploads/:uploadId`
- `GET /media/library`
- `GET|PATCH|DELETE /media/assets/:assetId`
- `POST /media/assets/:assetId/restore`
- `POST /media/assets/:assetId/share`
- `DELETE /media/assets/:assetId/shares/:messageId`
- `GET /media/assets/:assetId/content`
- `POST /media/albums`
- `POST /media/albums/:albumId/items`
- `DELETE /media/albums/:albumId/items/:assetId`

Content delivery is private/no-store, supports `206` byte ranges, sets `nosniff`, and uses RFC 5987-compatible UTF-8 file names.

## Windows And PDF Runtime

The desktop static server allows authenticated image/media requests only to loopback API ports. `.mjs` files are served as JavaScript so the bundled PDF.js module worker can execute under `nosniff`. PDF bytes are fetched with the current HttpOnly session and passed to PDF.js as an `ArrayBuffer`; no public object URL or third-party document viewer is used.

The installed renderer test uses a fresh temporary Electron user-data profile, claims the seed owner, uploads and shares an image, renders it through the authenticated endpoint, privately stores a PDF, checks a nonblank PDF canvas, opens the selected document in another Electron window, and verifies original/preview bytes under that isolated object root. The profile is removed after PostgreSQL and Electron stop, so test accounts and media cannot accumulate in the normal user database.

## Verification

`npm run media:integration` creates a fresh PostgreSQL database and proves:

- resumable checkpoints survive API restart;
- text chat succeeds while an upload is incomplete;
- part and final hash mismatch fail closed;
- MIME mismatch and malware fixtures are quarantined;
- private archive non-disclosure and exact hub recipient grants;
- guest preview/download policy and immediate revoke;
- authorized byte ranges and blocked unauthorized content;
- owner-only date/place metadata, indexed filters, albums, audit events, and restart restore;
- known Stage 4 pre-release migration reconciliation is narrowly verified and unknown checksum drift still aborts startup.

The pre-commit harness runs all earlier authentication, invitation, conversation, and contacts suites before this media suite and the base smoke check.
